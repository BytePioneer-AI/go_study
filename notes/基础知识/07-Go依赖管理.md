# 07 - Go 依赖管理

## 本节目标

这一节你先掌握 2 个核心问题：

1. Go 是怎么记录依赖的
2. 日常开发里你应该用哪些命令管理依赖

如果你写过 Node.js、Python、Java，这一节最重要的心智转换是：

**Go 的依赖管理以 `go.mod` 为中心，以模块版本为单位，而不是把依赖源码塞进每个项目目录。**

---

## 1. `go.mod` 是依赖管理的中心

先看一个典型的 `go.mod`：

```go
module go_study

go 1.25.5

require github.com/google/uuid v1.6.0
```

你可以把它理解成：

- `module`：当前项目的模块路径
- `go`：这个模块使用的 Go 语言版本基线
- `require`：当前项目依赖了哪些模块、什么版本

### 和其他语言对比

#### Node.js

你可能会想到 `package.json`：

- `dependencies` 记录依赖
- 但包通常会下载到项目里的 `node_modules/`

#### Python

你可能会想到 `requirements.txt` 或 `pyproject.toml`：

- 也会记录依赖版本
- 但通常依赖安装在虚拟环境里

#### Go

Go 的重点是：

- 项目里记录“需要什么”
- 依赖源码缓存在全局模块缓存里
- 当前项目不需要一个 `node_modules/` 目录

所以你先记一句：

**`go.mod` 记录依赖声明，真正的依赖内容通常不放在项目目录里。**

---

## 2. `go.sum` 不是第二个 `go.mod`

很多初学者会把 `go.sum` 和 `go.mod` 混在一起。

它们职责不同：

- `go.mod`：声明“我要依赖谁”
- `go.sum`：记录下载过的模块校验信息，保证依赖内容一致

你可以把 `go.sum` 理解成“依赖完整性校验清单”。

### 这和其他语言的对比

- Node.js 里像 `package-lock.json`
- Yarn 里像 `yarn.lock`
- Python 生态里没有完全一一对应，但作用上接近“锁定和校验”

### 初学阶段记住两个原则

1. `go.mod` 和 `go.sum` 都应该提交到版本控制
2. `go.sum` 一般不要手动改

---

## 3. Go 依赖是怎么加进来的

Go 很少让你手动编辑依赖文件。

典型流程是：

1. 你在代码里写 `import`
2. 执行构建、测试或整理命令
3. Go 工具链自动补全 `go.mod` / `go.sum`

比如你写了：

```go
import "github.com/google/uuid"
```

然后执行：

```bash
go mod tidy
```

Go 就会把对应模块写入 `go.mod`，并更新 `go.sum`。

### Go 的习惯用法

在很多语言里，你可能先“安装包”，再开始写代码。

Go 里更常见的是：

- 先写 import
- 再让工具链解析并整理依赖

这体现了 Go 的一个风格：

**依赖管理尽量交给工具链，而不是手工维护。**

---

## 4. 最常用的依赖命令

### `go mod tidy`

这是你最常用的命令。

作用：

- 补上代码里用到了但 `go.mod` 里缺失的依赖
- 删除 `go.mod` 里不再使用的依赖
- 更新 `go.sum`

```bash
go mod tidy
```

你可以把它理解成“整理依赖到最小准确集合”。

### `go get`

现在它主要用于：

- 添加依赖
- 升级依赖
- 指定依赖版本

例如：

```bash
go get github.com/google/uuid@latest
go get github.com/google/uuid@v1.6.0
```

### `go list -m all`

查看当前模块依赖了哪些模块：

```bash
go list -m all
```

### `go mod download`

预先把依赖下载到本地缓存：

```bash
go mod download
```

这在 CI 或网络不稳定时很常见。

---

## 5. 版本写法怎么看

在 Go 模块里，你最常见到的是：

```text
v1.2.3
v1.6.0
v0.9.1
```

这本质上也是语义化版本（SemVer）。

### 初学时先抓住 3 个点

1. `v1.x.x`：通常表示稳定主版本
2. `v0.x.x`：通常表示还在快速变化，兼容性承诺较弱
3. 升级大版本时要更小心

### Go 和其他语言的一个区别

在 Node.js 里，你经常看到：

- `^1.2.3`
- `~1.2.3`

Go 的 `go.mod` 里记录的是**确定版本**，不是这种范围表达式。

也就是说，Go 更强调：

**当前模块解析出来的是一个明确版本，而不是模糊区间。**

---

## 6. 为什么 Go 没有 `node_modules`

这是很多有前端背景的人最直观的差异。

Go 把依赖放在全局模块缓存里，常见位置可以通过下面命令看：

```bash
go env GOPATH
```

模块缓存一般在：

```text
<GOPATH>/pkg/mod
```

这样做的好处：

- 同一个版本只下载一次
- 多个项目共享缓存
- 项目目录更干净

你可以把它理解成：

**Go 更像“项目只声明依赖，工具链统一缓存依赖”。**

---

## 7. 日常开发中的推荐心智模型

你现在可以先用下面这套最简流程：

### 场景 A：新增第三方依赖

1. 在代码里写 `import`
2. 执行：

```bash
go mod tidy
```

### 场景 B：想升级某个依赖

执行：

```bash
go get github.com/google/uuid@latest
go mod tidy
```

### 场景 C：删掉某个依赖后想清理

执行：

```bash
go mod tidy
```

### 一个很重要的习惯

如果你修改了 import、增加了库、删除了库，最后都可以跑一次：

```bash
go mod tidy
```

它基本就是 Go 依赖管理里的“收尾整理动作”。

---

## 8. 一个最小示例

下面是一个使用第三方依赖的最小例子。

`main.go`

```go
package main

import (
	"fmt"

	"github.com/google/uuid"
)

func main() {
	id := uuid.New()
	fmt.Println("new id:", id)
}
```

如果这是你第一次引入这个包，常见操作是：

```bash
go mod tidy
go run .
```

### 你应该关注什么

- import 的是包路径
- `go.mod` 会新增 `require` 项
- `go.sum` 会新增校验记录

## 9. 初学者最容易踩的坑

### 坑 1：手动乱改 `go.sum`

不推荐。

`go.sum` 是工具链维护的，不是给你手写的。

### 坑 2：以为删了 import 就自动删依赖

通常要再执行：

```bash
go mod tidy
```

才能把不需要的依赖从模块文件里清理掉。

### 坑 3：把 Go 依赖管理想成 `pip install` 或 `npm install`

Go 不是完全没有“安装”概念，但在项目依赖管理这件事上，更推荐你围绕：

- `import`
- `go mod tidy`
- `go get`

来理解。

### 坑 4：把包（package）和模块（module）混为一谈

这是非常常见的。

你先这样区分：

- `package`：代码组织单位，通常对应一个目录
- `module`：依赖管理单位，由 `go.mod` 定义

一个 module 里可以有多个 package。

## 练习题

目标：

理解 `go.mod`、`go.sum`、第三方依赖引入流程，以及 module 和 package 的区别。

要求：

1. 在当前目录的 `main.go` 中引入 `github.com/google/uuid`
2. 在 `main()` 中生成一个新的 UUID 并打印
3. 自己执行 `go mod tidy`
4. 观察 `go.mod` 和 `go.sum` 的变化
5. 再尝试删除 `uuid` 的 import 和使用代码，然后再次执行 `go mod tidy`
6. 观察依赖是否被清理

### 参考答案代码

`main.go`

```go
package main

import (
	"fmt"

	"github.com/google/uuid"
)

func main() {
	id := uuid.New()
	fmt.Println("generated uuid:", id)
}
```

### 预期运行结果

输出会类似下面这样，但 UUID 每次都不同：

```text
generated uuid: 3f29f985-6d89-4f7a-9b58-9d4b4f66c2a1
```

---

## 本节小结

你现在先掌握这几个结论就够了：

1. `go.mod` 负责声明模块和依赖
2. `go.sum` 负责记录依赖校验信息
3. Go 依赖通常通过 `import` + 工具链命令维护
4. 日常最常用的整理命令是 `go mod tidy`
5. `package` 和 `module` 不是一回事
