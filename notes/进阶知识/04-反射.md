# 04 - 反射

## 本节目标

这一节先掌握 2 个最核心的问题：

1. Go 的反射是什么，为什么平时不常手写它
2. `reflect.Type` 和 `reflect.Value` 分别代表什么

如果你之前写过 Java、C#、Python，这一节最重要的心智转换是：

**Go 有反射，但它不是日常业务代码的默认工具，而是“确实需要处理未知类型时”才拿出来的工具。**

---

## 1. 反射是什么

先说直白一点：

反射就是 **程序在运行时检查值的类型信息，并且按这个类型信息做操作**。

比如你手里只有一个 `any`（也就是接口值），但你运行时想知道：

- 它到底是不是 `struct`
- 它的类型名是什么
- 它有哪些字段
- 某个字段当前值是多少

这时候就会用到反射。

### 和其他语言的对比

#### Java / C#

你可能见过这种思路：

- `obj.getClass()`
- 枚举字段、方法
- 框架自动注入、自动序列化

#### Python

你可能更习惯：

- `type(x)`
- `getattr`
- `setattr`
- `hasattr`

#### Go

Go 也能做类似事情，但风格上更克制。

Go 的态度通常是：

- **能用静态类型解决，就不要上反射**
- 只有在“类型在编译期不知道”的时候，才考虑 `reflect`

这很符合 Go 一贯的设计哲学：

**先要简单、可读、可维护，再追求动态能力。**

---

## 2. 为什么 Go 平时不鼓励滥用反射

因为反射虽然灵活，但代价也很明显：

1. **代码更难读**：本来字段访问是 `u.Name`，用了反射后就变成字符串和运行时判断
2. **更容易出错**：很多错误变成运行时才发现
3. **性能更差**：比直接访问字段慢
4. **失去类型系统帮助**：IDE 补全、编译器检查都会变弱

所以你可以先记一句话：

**反射不是“更高级的写法”，而是“不得不用时的写法”。**

常见使用场景通常是：

- JSON / ORM / 配置加载 这类框架代码
- 通用库
- 调试或打印未知结构

普通业务代码里，优先考虑：

- 明确类型
- 接口
- 普通函数调用

---

## 3. 反射入口：`reflect.TypeOf` 和 `reflect.ValueOf`

Go 反射最常见的入口有两个：

```go
reflect.TypeOf(x)
reflect.ValueOf(x)
```

- `reflect.TypeOf(x)`：拿到类型信息
- `reflect.ValueOf(x)`：拿到值本身的反射表示

先看一个最小例子：

```go
package main

import (
	"fmt"
	"reflect"
)

func main() {
	name := "gopher"

	t := reflect.TypeOf(name)
	v := reflect.ValueOf(name)

	fmt.Println(t)
	fmt.Println(v)
}
```

这里：

- `t` 更关注“它是什么类型”
- `v` 更关注“这个值本身是什么”

你可以先粗略理解成：

- `Type` 像“类型说明书”
- `Value` 像“运行时拿在手里的那个值”

---

## 4. `reflect.Type`：看类型信息

看例子：

```go
package main

import (
	"fmt"
	"reflect"
)

type User struct {
	Name string
	Age  int
}

func main() {
	u := User{Name: "Tom", Age: 20}
	t := reflect.TypeOf(u)

	fmt.Println("类型:", t)
	fmt.Println("种类:", t.Kind())
	fmt.Println("类型名:", t.Name())
}
```

这里你会看到几个概念：

### `t.Name()`

表示这个具体类型的名字，比如 `User`。

### `t.Kind()`

表示这个值底层属于哪一类。

比如：

- `int`
- `string`
- `struct`
- `slice`
- `map`
- `ptr`

这两个概念很容易混：

- `Name()` 更像“这个类型叫啥”
- `Kind()` 更像“它本质属于哪一大类”

例如：

`User` 的：
- `Name()` 是 `User`
- `Kind()` 是 `struct`

这个区别很重要，因为很多反射代码会先判断：

```go
if t.Kind() == reflect.Struct {
	// 按结构体处理
}
```

---

## 5. `reflect.Value`：看值、取字段值

如果 `Type` 是“说明书”，那 `Value` 就是“实际东西”。

看例子：

```go
package main

import (
	"fmt"
	"reflect"
)

type User struct {
	Name string
	Age  int
}

func main() {
	u := User{Name: "Tom", Age: 20}
	v := reflect.ValueOf(u)

	fmt.Println("值:", v)
	fmt.Println("字段数量:", v.NumField())
	fmt.Println("第一个字段:", v.Field(0))
	fmt.Println("第二个字段:", v.Field(1))
}
```

这里因为 `u` 是结构体，所以可以：

- `v.NumField()` 看字段数
- `v.Field(i)` 取第 `i` 个字段

这说明反射可以在“事先不知道具体字段名”的情况下，遍历结构体内容。

这也是很多框架能自动处理结构体的基础。

---

## 6. 一个关键区别：接口里的值，反射看到的是“动态类型”

这和你前面学过的接口很有关系。

看例子：

```go
package main

import (
	"fmt"
	"reflect"
)

type User struct {
	Name string
}

func main() {
	var x any = User{Name: "Tom"}

	fmt.Println(reflect.TypeOf(x))
}
```

虽然 `x` 的静态类型是 `any`，但运行时里面装的是 `User`。

所以 `reflect.TypeOf(x)` 看到的是：

**接口里实际装着的动态类型。**

这一点非常关键，因为反射几乎总是在处理接口值。

你可以把它和接口章节连起来理解：

- 接口让你“统一接住不同类型”
- 反射让你“运行时再看接口里到底装了什么”

---

## 7. 指针和反射：经常会遇到 `ptr`

Go 反射里另一个非常常见的坑是：

你以为拿到的是结构体，结果看到的是指针。

```go
package main

import (
	"fmt"
	"reflect"
)

type User struct {
	Name string
}

func main() {
	u := User{Name: "Tom"}

	t1 := reflect.TypeOf(u)
	t2 := reflect.TypeOf(&u)

	fmt.Println(t1.Kind())
	fmt.Println(t2.Kind())
}
```

输出会是：

- `t1.Kind()` 是 `struct`
- `t2.Kind()` 是 `ptr`

也就是说：

- 值本身是结构体 → `struct`
- 指向结构体的指针 → `ptr`

如果你真想看指针指向的元素，要继续取一层。

在 `Type` 上常用：

```go
t.Elem()
```

例如：

```go
package main

import (
	"fmt"
	"reflect"
)

type User struct {
	Name string
}

func main() {
	u := User{Name: "Tom"}
	t := reflect.TypeOf(&u)

	fmt.Println(t.Kind())
	fmt.Println(t.Elem().Kind())
	fmt.Println(t.Elem().Name())
}
```

这个现象你要先建立敏感度：

**反射里很多判断失败，不是因为不是结构体，而是因为你拿到的是“指向结构体的指针”。**

## 8. 遍历结构体字段示例

下面给你一个稍微完整、但仍然容易读懂的例子：

```go
package main

import (
	"fmt"
	"reflect"
)

type User struct {
	Name string
	Age  int
}

func main() {
	u := User{Name: "Tom", Age: 20}
	v := reflect.ValueOf(u)
	t := reflect.TypeOf(u)

	for i := 0; i < v.NumField(); i++ {
		fieldType := t.Field(i)
		fieldValue := v.Field(i)

		fmt.Printf("字段名: %s, 字段类型: %s, 字段值: %v\n", fieldType.Name, fieldType.Type, fieldValue)
	}
}
```

这里体现出反射的典型模式：

1. 先拿 `Type`
2. 再拿 `Value`
3. 用索引遍历字段
4. 同时读“字段定义信息”和“字段当前值”

这种模式你在很多库源码里都会看到。

---

## 9. 当前阶段你最该掌握的，不是“会写复杂反射”，而是“会读”

对于刚学 Go 的阶段，我更建议你先做到：

1. 看见 `reflect.TypeOf` / `reflect.ValueOf` 不害怕
2. 能读懂 `Kind()`、`Name()`、`Field()`、`Elem()` 这些基本操作
3. 明白为什么这里不用普通字段访问，而要用反射

因为在实际开发里，你更常见的情况是：

- 在标准库或第三方库源码里读到反射
- 排查框架为什么没识别某个结构体字段
- 理解为什么传值和传指针会行为不同

也就是说，**现阶段“能理解反射代码”比“自己大量写反射代码”更重要。**

## 10. 先记住这几个高频 API

这一节先只记最常见的几个：

- `reflect.TypeOf(x)`
- `reflect.ValueOf(x)`
- `t.Kind()`
- `t.Name()`
- `t.Elem()`
- `v.NumField()`
- `v.Field(i)`

先别一次记太多。

你现在只要能回答下面几个问题，就说明入门了：

- 什么时候需要反射？
- `Type` 和 `Value` 区别是什么？
- 为什么 `&u` 的 `Kind()` 是 `ptr`？
- 为什么很多反射代码一上来先判断 `Kind()`？

---

## 本节小结

这一节你先建立这几个心智模型：

1. **反射是运行时查看和操作类型信息的工具**
2. **Go 不鼓励把反射当日常写法，只有在类型未知时才用它**
3. **`reflect.Type` 看类型，`reflect.Value` 看值**
4. **反射里要特别注意值和指针的区别**

如果你把这 4 点吃透，已经够支撑你读很多基础反射代码了。

## 练习题

这次给你一个小练习，目标是“只读反射信息”，先不要做修改字段这种更难的操作。

### 题目

在当前目录新建一个 Go 文件，定义一个 `Book` 结构体：

- `Title string`
- `Pages int`

然后在 `main` 函数里：

1. 创建一个 `Book` 值
2. 用 `reflect.TypeOf` 打印它的类型名和 kind
3. 用 `reflect.ValueOf` 遍历并打印每个字段的：
   - 字段名
   - 字段类型
   - 字段值

### 验收标准

运行后，输出里至少能看出：

- `Book` 这个类型名
- `struct` 这个 kind
- `Title` 和 `Pages` 两个字段的信息

### 额外思考题

把：

```go
reflect.TypeOf(book)
```

改成：

```go
reflect.TypeOf(&book)
```

观察 `Kind()` 有什么变化，并解释为什么。
